## 4. Cross-Repository Findings

### 4.1 Durable global strengths

The ecosystem consistently demonstrates:

- repository-backed source-of-truth discipline;
- separation of research, proposal, decision, implementation, validation, approval, and owner override;
- current-state handoffs for continuity;
- explicit stage or phase gates;
- recovery planning before risky work;
- privacy-aware and local-first preferences;
- preservation of source assets and unrelated work;
- evidence-based completion reports;
- reusable agent, prompt, and documentation structures;
- project-specific boundaries that prevent one project's stack from becoming a universal requirement.

### 4.2 Capability calibration

The public profile may describe capabilities only at the level supported by evidence.

Use:

- `IMPLEMENTED` when code or a deliverable exists;
- `VALIDATED` when relevant checks or runtime evidence exist;
- `ACTIVELY DEVELOPING` for unfinished implementation;
- `DOCUMENTED` for repository-backed plans and operating systems;
- `EXPLORING` for research, experiments, and minimally initialized projects.

Do not use a repository name, dependency, plan, prompt, or empty folder as evidence of a completed product.

### 4.3 Reusable workflow lessons

The strongest reusable lessons are:

1. Read repository truth before using chat memory.
2. Maintain one concise active handoff instead of copying full project history into every state file.
3. Preserve decision history and mark superseded directions explicitly.
4. Keep owner continuation decisions separate from technical PASS.
5. Make source assets immutable by default.
6. Use the smallest appropriate tool for routine work and reserve coding agents for real implementation.
7. Require actual output evidence before claiming successful rendering, builds, exports, or document generation.
8. Promote generic structures globally while keeping project-only technology and risk local.
9. Treat public-profile synchronization as a privacy review, not merely a documentation update.
